# 方案三：嵌套集（Nested Set Model）

> 给每个节点一对 `(lft, rgt)` 左右值，按 DFS 先序遍历分配。**子树关系等价于"左右值落在某个区间内"**。读极快，写极慢，是历史悠久的"读多写少"经典方案。

---

## 1. 核心思想

想象你拿着一支笔，沿着树的轮廓走一圈：

- 第一次进入某个节点（向下进入）时记录 `lft`；
- 最后离开这个节点（向上回退）时记录 `rgt`；
- 编号一路自增，永不回退。

走完一圈之后，**每个节点都有一对 `(lft, rgt)`**，并且：

> 节点 `A` 是节点 `B` 的祖先  ⇔  `A.lft < B.lft AND A.rgt > B.rgt`

这就是嵌套集的全部精髓——把"祖先关系"翻译成了"区间包含关系"。

举例：

```
            (1, 14)
              A
             / \
       (2, 7)   (8, 13)
          B       C
         / \      |
   (3, 4) (5, 6) (9, 12)
      D     E       F
                    |
                 (10, 11)
                    G
```

观察：

- `A` 是所有人的祖先：`A.lft = 1 < 任何人的 lft`，`A.rgt = 14 > 任何人的 rgt`。
- `F.lft = 9, F.rgt = 12`，区间内只包含 `(10, 11)` → `F` 只有一个后代 `G`。
- 一个节点的子节点数 = `(rgt - lft - 1) / 2`。
- 整棵树的节点数 = `(根.rgt - 根.lft + 1) / 2 = (14 - 1 + 1) / 2 = 7`。

## 2. 表结构

### 2.1 标准 DDL

```sql
CREATE TABLE tree_node (
    id      BIGSERIAL    PRIMARY KEY,
    tree_id BIGINT       NOT NULL,
    lft     INT          NOT NULL,
    rgt     INT          NOT NULL,
    name    VARCHAR(255) NOT NULL,
    payload JSONB,

    CONSTRAINT chk_lft_lt_rgt CHECK (lft < rgt),
    CONSTRAINT uniq_tree_lft  UNIQUE (tree_id, lft),
    CONSTRAINT uniq_tree_rgt  UNIQUE (tree_id, rgt)
);

CREATE INDEX idx_tree_lft ON tree_node (tree_id, lft);
CREATE INDEX idx_tree_rgt ON tree_node (tree_id, rgt);
```

### 2.2 推荐的"加强版"

纯嵌套集模型有几个老大难问题：直接孩子查询不友好、看不出深度、没有 `parent_id` 不便于 ORM。所以工程实践里几乎都是"嵌套集 + 邻接表 + 深度"的混合：

```sql
CREATE TABLE tree_node (
    id        BIGSERIAL    PRIMARY KEY,
    tree_id   BIGINT       NOT NULL,
    parent_id BIGINT       NULL,                 -- 邻接表冗余
    depth     INT          NOT NULL,             -- 深度冗余
    lft       INT          NOT NULL,
    rgt       INT          NOT NULL,
    name      VARCHAR(255) NOT NULL,
    payload   JSONB,

    CONSTRAINT chk_lft_lt_rgt CHECK (lft < rgt),
    CONSTRAINT fk_parent FOREIGN KEY (parent_id) REFERENCES tree_node(id)
);

CREATE INDEX idx_tree_lft       ON tree_node (tree_id, lft);
CREATE INDEX idx_tree_rgt       ON tree_node (tree_id, rgt);
CREATE INDEX idx_tree_parent    ON tree_node (tree_id, parent_id);
```

加 `parent_id`、`depth` 之后空间多了一点，但**所有缺点都被治好了**——这是真正实用的嵌套集形态。

### 2.3 为什么不能用普通外键防环

嵌套集用 `(lft, rgt)` 编码祖先关系，**根本没有"父"这个字段**（或者说"父"是用区间包含算出来的）。所以传统的"自引用外键"放不上去——这就是它一致性弱的原因。

## 3. 增删改查全套 SQL

### 3.1 取整棵子树（嵌套集的杀手级操作）

```sql
-- 用同表自连接：父节点的区间包含子节点的区间
SELECT child.*
FROM tree_node parent
JOIN tree_node child
  ON parent.tree_id = child.tree_id
 AND child.lft BETWEEN parent.lft AND parent.rgt
WHERE parent.id = :id;
```

**没有递归、没有 CTE、零跳转**——一次 BETWEEN 走 B-Tree 索引就出来了。

### 3.2 取祖先链

```sql
SELECT ancestor.*
FROM tree_node node
JOIN tree_node ancestor
  ON node.tree_id = ancestor.tree_id
 AND node.lft BETWEEN ancestor.lft AND ancestor.rgt
WHERE node.id = :id
ORDER BY ancestor.lft;
```

最后一行按 `lft` 升序就是从根到自己的顺序。

### 3.3 取直接孩子

纯嵌套集做这件事很难（要"区间包含但中间不能再包含别人"），需要复杂的 NOT EXISTS 子查询。**这就是为什么必须冗余 `parent_id`**：

```sql
SELECT * FROM tree_node WHERE parent_id = :id;
```

### 3.4 取节点深度

```sql
-- 不冗余 depth 时只能现算
SELECT COUNT(*) - 1 AS depth
FROM tree_node ancestor
JOIN tree_node node ON node.lft BETWEEN ancestor.lft AND ancestor.rgt
WHERE node.id = :id AND ancestor.tree_id = node.tree_id;

-- 冗余了 depth 后
SELECT depth FROM tree_node WHERE id = :id;
```

### 3.5 取叶子

```sql
SELECT * FROM tree_node WHERE rgt = lft + 1;
```

`rgt = lft + 1` 等价于"区间里不可能再塞下任何子节点"，所以这是叶子。这是嵌套集少有的"令人愉悦"的小巧查询。

## 4. 复杂操作：插入 / 删除 / 移动

嵌套集的所有写操作都遵循一个原则：**整体右移 / 左移 / 平移**。

### 4.1 插入新节点（作为某节点的最后一个孩子）

要在父节点 `P` 下挂一个新节点 `N`：

1. 在 `P.rgt` 这个位置"撑开"一个宽度为 2 的空隙；
2. 把 `N.lft` / `N.rgt` 填进这个空隙。

```sql
BEGIN;

-- 1. 把所有 rgt >= P.rgt 的节点向右挪 2
UPDATE tree_node
SET rgt = rgt + 2
WHERE tree_id = :tree_id AND rgt >= (SELECT rgt FROM tree_node WHERE id = :parent_id);

-- 2. 把所有 lft >  P.rgt 的节点向右挪 2
UPDATE tree_node
SET lft = lft + 2
WHERE tree_id = :tree_id AND lft >  (SELECT rgt FROM tree_node WHERE id = :parent_id);

-- 注意：上面两条 UPDATE 会破坏 P.rgt 自身的旧值，需要先把 P.rgt 缓存到变量里再用
-- 3. 插入新节点
INSERT INTO tree_node (tree_id, parent_id, depth, lft, rgt, name)
SELECT tree_id, id, depth + 1, rgt - 2, rgt - 1, :name
FROM tree_node WHERE id = :parent_id;

COMMIT;
```

**最坏复杂度 O(N)**，整张表都可能被改写。

### 4.2 删除节点（连同子树）

1. 拿到子树的 `(lft, rgt)`；
2. 删掉所有落在区间内的节点；
3. 把"右半边"整体左移一段距离，距离 = `rgt - lft + 1`。

```sql
BEGIN;

WITH target AS (SELECT lft, rgt FROM tree_node WHERE id = :id)
DELETE FROM tree_node
WHERE tree_id = :tree_id
  AND lft BETWEEN (SELECT lft FROM target) AND (SELECT rgt FROM target);

UPDATE tree_node
SET rgt = rgt - (SELECT rgt - lft + 1 FROM target),
    lft = CASE WHEN lft > (SELECT rgt FROM target)
               THEN lft - (SELECT rgt - lft + 1 FROM target)
               ELSE lft
          END
WHERE tree_id = :tree_id
  AND rgt > (SELECT rgt FROM target);

COMMIT;
```

注意 `target` 必须先取出来缓存，因为子查询里它们随后会被改写。

### 4.3 移动子树（最痛的操作）

把以 `n` 为根的子树整体挪到新父亲 `m` 下：

1. 计算子树宽度 `width = n.rgt - n.lft + 1`；
2. 计算偏移量 `delta = m.rgt - n.lft`；
3. 把子树先"挪出"原位置（暂存到一个临时区间）；
4. 把右半边节点向左压缩 `width`；
5. 把目标位置右半边节点向右扩展 `width`；
6. 把暂存的子树"塞回"目标位置。

完整 SQL 非常啰嗦（少说也是 5 ~ 6 条 UPDATE），写错一处就把整棵树打烂。**生产环境一般会把这段逻辑封装到存储过程或应用层 helper 里**，绝不在业务代码里裸写。

参考 PostgreSQL 实现思路：

```sql
-- 伪代码版本
function move_subtree(n_id, new_parent_id):
  width  = n.rgt - n.lft + 1
  -- 1. 把子树标记为负数（暂存）
  UPDATE tree_node SET lft = -lft, rgt = -rgt
  WHERE lft BETWEEN n.lft AND n.rgt;

  -- 2. 关闭旧位置的空隙
  UPDATE tree_node SET lft = lft - width WHERE lft > n.rgt;
  UPDATE tree_node SET rgt = rgt - width WHERE rgt > n.rgt;

  -- 3. 在新位置撑开空隙（用刷新过的 m.rgt）
  UPDATE tree_node SET lft = lft + width WHERE lft > m.rgt;
  UPDATE tree_node SET rgt = rgt + width WHERE rgt >= m.rgt;

  -- 4. 把暂存的子树（负数）平移回正确位置
  delta = m.rgt - n.lft
  UPDATE tree_node SET lft = -lft + delta, rgt = -rgt + delta
  WHERE lft < 0;
```

## 5. 性能分析

| 操作 | 复杂度 | 备注 |
| --- | --- | --- |
| 取子树 | O(子树大小) | 一次 BETWEEN，零 JOIN |
| 取祖先 | O(深度) | 一次自连接 |
| 取叶子 | O(叶子数) | `rgt = lft + 1` |
| 取直接孩子 | 难 / 慢 | 需要 NOT EXISTS 或额外 `parent_id` |
| 取深度 | 难 / 慢 | 需要 COUNT 或额外 `depth` |
| 插入节点 | O(N) | 需要平移半棵树的左右值 |
| 删除节点 | O(N) | 同上 |
| 移动子树 | O(N) | 4~6 条 UPDATE 串起来 |

**关键结论**：嵌套集的读极快、写极慢。它适合"几乎只读"的场景，一旦写入压力上来就会崩。

---

## 6. 并发与一致性

嵌套集的写操作有一个非常严重的"全表锁"问题：

- 一次插入 / 删除 / 移动通常都要修改半张表的左右值；
- 任意两个并发写都极易冲突（更新同一批行）；
- 必须用**串行隔离级别**或**悲观锁**保证一致性。

工程上的应对：

1. 写操作加表级锁 `LOCK TABLE tree_node IN EXCLUSIVE MODE`；
2. 写入路径走单线程串行队列；
3. 写多的时候根本就不应该选嵌套集。

并发不是嵌套集的"小毛病"，是它的**结构性短板**。

## 7. 典型陷阱

### 7.1 左右值溢出

如果 `lft` / `rgt` 用 `INT`（4 字节，最大约 21 亿），节点数特别多 + 频繁插入会消耗左右值空间。可以：

- 用 `BIGINT`；
- 或者在写入达到一定密度时**重排**整棵树（把 `(lft, rgt)` 重新按 1, 2, 3... 编号）。

### 7.2 没有 `parent_id` / `depth` 时几乎没法用

直接孩子、深度都查不动。**新项目用嵌套集请一律加上 `parent_id` + `depth` 冗余列。**

### 7.3 多棵树共表时务必加 `tree_id`

嵌套集的查询全靠区间，没有 `tree_id` 过滤会把所有树的区间混在一起算，结果完全错。

### 7.4 写入顺序敏感

`UPDATE` 子查询里如果引用了正在被改的列（如 `P.rgt`），子查询会读到"中间状态"，结果错乱。**必须先把所需值取到变量 / WITH 里**再用。

### 7.5 拷贝整棵树到另一棵树

最佳做法：先按 BFS 建一份新结构，重新编号 `lft` / `rgt`，再批量插入。**不要**试图直接复制 `(lft, rgt)` 然后整体加一个偏移——容易和目标树原有节点冲突。

## 8. 优缺点

### 8.1 优点

- **取子树极快**：所有方案里读子树最快的之一，且不依赖递归 CTE。
- **取祖先链一次自连接**。
- **判断"是否后代"**：`A.lft BETWEEN B.lft AND B.rgt`，几乎免费。
- 在 90 年代 / 2000 年代初极度流行，是经典 SQL 技巧的代表。

### 8.2 缺点

- **写极慢**：任何结构改动都要平移半棵树。
- **并发性极差**：必须串行写。
- **缺乏强一致约束**：左右值的协调完全靠应用层。
- **直接孩子 / 深度 不冗余就查不动**。
- **思维负担重**：对开发者不友好，容易写错。

---

## 9. 适用场景

适合用嵌套集的场景：

1. **静态或几乎静态的树**：导入一次后基本只读，例如：
   - 商品类目树（按月才更新一次）；
   - 行政区划（国 / 省 / 市 / 县）；
   - 生物分类（界门纲目科属种）。
2. **以"取整棵子树"或"判断祖先关系"为绝对热路径**的查询。
3. **写入操作可以离线批处理**：晚上低峰期统一刷新左右值。

不适合的场景：

- 任何"用户自己拖拽节点"的协作类应用。
- 频繁挂载 / 移除子树的业务（菜单、组织架构调整）。
- 高并发写入。

## 10. 与 `tree.py` 的映射

| 内存模型 | 嵌套集列 |
| --- | --- |
| `TreeNode.value` | `payload` |
| `TreeNode.name` | `name` |
| `TreeNode.parent` | `parent_id`（冗余） |
| `TreeNode.children` | 用 `parent_id` 查（冗余）；纯嵌套集需要复杂 NOT EXISTS |
| `TreeNode.level` | `depth`（冗余） |
| `TreeNode.order` | **几乎和 `lft` 同义**——`lft` 升序就是 DFS 先序 |
| `Tree.root` | `lft = MIN(lft)` 或 `parent_id IS NULL` |

### 10.1 把 `tree.py` 的树落到嵌套集

实现思路：用 `Tree.dfs()` 遍历一次，按出入栈给每个节点分配 `lft` / `rgt`：

```python
def to_nested_set(tree: Tree) -> list[dict]:
    rows = []
    counter = [0]

    def visit(node: TreeNode):
        counter[0] += 1
        lft = counter[0]
        for child in node.children:
            visit(child)
        counter[0] += 1
        rgt = counter[0]
        rows.append({
            "id":      id(node),
            "name":    node.name,
            "depth":   node.level,
            "lft":     lft,
            "rgt":     rgt,
        })

    if tree.root is not None:
        visit(tree.root)
    return rows
```

注意嵌套集的 `lft` 编号顺序和 `tree.py` 里 `order` 字段（叶子按 DFS 全局编号）的语义不完全相同，但**排序结果是一致的**——按 `lft` 排序等价于按 DFS 先序遍历。

### 10.2 反向加载

```python
def load_nested_set(rows: list[dict]) -> Tree:
    rows.sort(key=lambda r: r["lft"])
    tree = Tree()
    stack: list[tuple[TreeNode, int]] = []   # (node, rgt)
    for r in rows:
        node = TreeNode(value=r["payload"], name=r["name"])
        while stack and stack[-1][1] < r["lft"]:
            stack.pop()
        if stack:
            parent = stack[-1][0]
            node.parent = parent
            parent.children.append(node)
        else:
            tree.root = node
        stack.append((node, r["rgt"]))
    if tree.root is not None:
        tree.refresh_levels()
        tree.refresh_orders()
    return tree
```

按 `lft` 排序后用栈维护"当前还没闭合的祖先"，O(N) 即可重建整棵树。

---

## 11. 一句话总结

> **嵌套集是"为读而生"的方案：取子树 / 判祖先快到极致，但任何写入都要平移半棵树。** 如果你的树几乎只读、写入可以离线批量做，它仍然是个不错的选择；其他场景下，[闭包表](./方案04-闭包表.md) 几乎能在任何维度上代替它。
