# 方案四：闭包表（Closure Table）

> 把"祖先-后代"关系从节点表里**抽出来**，单独存到一张关系表里。每条记录形如 `(ancestor, descendant, depth)`，所有跨层级查询都退化成一次普通 JOIN。综合最优，工业界中大型系统的首选。

---

## 1. 核心思想

前三种方案都试图"在节点行内编码祖先关系"——要么靠 `parent_id` 间接推断，要么靠 `path` / `(lft, rgt)` 直接编码。闭包表则反过来：

> **节点和关系彻底分家**：节点表只存自己，关系表把"我是谁的后代、谁是我的祖先"统统列出来。

具体做法是：对每个节点，把"它和所有祖先的对应关系"全部展开存进一张关系表 `tree_closure`。

举例：树 `1 → 2 → 5`，闭包表里会出现这些行：

| ancestor | descendant | depth |
| --- | --- | --- |
| 1 | 1 | 0 |
| 2 | 2 | 0 |
| 5 | 5 | 0 |
| 1 | 2 | 1 |
| 2 | 5 | 1 |
| 1 | 5 | 2 |

注意几个要点：

- 每个节点都有一条 `(self, self, 0)` 的"自反"记录——这让"取子树包不包括自己"的边界条件统一了。
- `depth = 1` 就是直接父子关系。
- 一棵深度为 `D`、节点数为 `N` 的树，闭包表行数大约是 `O(N × 平均深度)`，最坏 `O(N × D)`。

---

## 2. 表结构

### 2.1 标准 DDL

```sql
-- 节点表（朴素的邻接表，作为真值来源）
CREATE TABLE tree_node (
    id        BIGSERIAL    PRIMARY KEY,
    tree_id   BIGINT       NOT NULL,
    parent_id BIGINT       NULL,
    name      VARCHAR(255) NOT NULL,
    payload   JSONB,
    created_at TIMESTAMPTZ NOT NULL DEFAULT now(),

    CONSTRAINT fk_parent FOREIGN KEY (parent_id) REFERENCES tree_node(id) ON DELETE CASCADE
);

CREATE INDEX idx_tree_parent ON tree_node (tree_id, parent_id);

-- 闭包关系表
CREATE TABLE tree_closure (
    ancestor   BIGINT NOT NULL,
    descendant BIGINT NOT NULL,
    depth      INT    NOT NULL,

    PRIMARY KEY (ancestor, descendant),
    CONSTRAINT fk_anc FOREIGN KEY (ancestor)   REFERENCES tree_node(id) ON DELETE CASCADE,
    CONSTRAINT fk_des FOREIGN KEY (descendant) REFERENCES tree_node(id) ON DELETE CASCADE,
    CONSTRAINT chk_depth_nonneg CHECK (depth >= 0)
);

CREATE INDEX idx_closure_descendant ON tree_closure (descendant);
CREATE INDEX idx_closure_depth      ON tree_closure (descendant, depth);
```

### 2.2 索引设计的细节

闭包表上有两个绝对不能少的索引：

1. `(ancestor, descendant)` 主键 → 加速"从祖先看后代"。
2. `(descendant)` 单列索引 → 加速"从后代看祖先"。

如果还经常查"某节点的直接孩子 / 直接父亲"，再加一个 `(ancestor, depth)` 或 `(descendant, depth)` 复合索引。

### 2.3 是否真的需要节点表里保留 `parent_id`

强烈建议保留。原因：

- `parent_id` 是写入时的真值来源；闭包表的所有行都从它推导而来。
- ORM、外键约束、级联删除等都依赖 `parent_id`。
- "查直接父亲 / 直接孩子"用 `parent_id` 一次完成，无需 JOIN 闭包表。

闭包表只是"为了加速跨层级查询而构建的派生索引"，不应该让它代替主存储。

### 2.4 本仓库 (`db.py`) 实际生成的 SQLite Schema

上文 2.1 给出的是 PostgreSQL 风格的"教学版"DDL。本仓库 (`ClosureTreeDB`) 跑在 SQLite 上，并且做了两处工程化扩展：

1. `tree_node` 增加两列 **`sibling_order` / `node_order`**，把"同父下顺序"和"全树 DFS 序号"也落库；
2. 新增一张 **`tree_index`**（`tree_id ↔ 根节点名`）的反查表，配套触发器自动维护（见 6.2.1）。

#### 表 / 字段速览

**`tree_node`**——节点表（邻接表，真值来源）：

| 字段 | 类型 | 约束 / 默认 | 含义 |
| --- | --- | --- | --- |
| `id` | `INTEGER` | `PRIMARY KEY AUTOINCREMENT` | 节点主键 |
| `tree_id` | `INTEGER` | `NOT NULL` | 所属树 ID（多树共表的分区键） |
| `parent_id` | `INTEGER` | `NULL`；`FK → tree_node(id) ON DELETE CASCADE` | 父节点 id；`NULL` 即根节点 |
| `name` | `TEXT` | `NOT NULL` | 节点名（业务可读，不要求全表唯一） |
| `payload` | `TEXT` | 可空 | 业务负载（JSON 文本，由代码层 `json.dumps`） |
| `sibling_order` | `INTEGER` | `NOT NULL DEFAULT 0` | 同一父亲下的兄弟序号 |
| `node_order` | `INTEGER` | `NOT NULL DEFAULT 0` | 全树 DFS 序号；与 `tree.py` 的 `TreeNode.order` 同语义；由 `_recalc_order(tree_id)` 在结构变更后整棵刷新 |
| `created_at` | `TEXT` | `NOT NULL DEFAULT (datetime('now'))` | 创建时间（ISO8601 文本） |

**`tree_closure`**——闭包表（祖先-后代派生索引）：

| 字段 | 类型 | 约束 / 默认 | 含义 |
| --- | --- | --- | --- |
| `ancestor` | `INTEGER` | `NOT NULL`；联合主键；`FK → tree_node(id) ON DELETE CASCADE` | 祖先节点 id |
| `descendant` | `INTEGER` | `NOT NULL`；联合主键；`FK → tree_node(id) ON DELETE CASCADE` | 后代节点 id |
| `depth` | `INTEGER` | `NOT NULL CHECK (depth >= 0)` | 祖先到后代的跳数；`0` = 自反、`1` = 直接父子 |

主键 = `(ancestor, descendant)`；闭包行由触发器全自动维护，业务代码不直接写。

**`tree_index`**——`tree_id ↔ 根节点名` 反查表：

| 字段 | 类型 | 约束 / 默认 | 含义 |
| --- | --- | --- | --- |
| `tree_id` | `INTEGER` | `PRIMARY KEY` | 树 ID（与 `tree_node.tree_id` 一一对应） |
| `root_name` | `TEXT` | `NOT NULL UNIQUE` | 该树根节点的 `name`，可按名反查 `tree_id` |
| `updated_at` | `TEXT` | `NOT NULL DEFAULT (datetime('now'))` | 最近一次维护时间 |

#### 索引一览

| 索引 | 表 | 列 | 用途 |
| --- | --- | --- | --- |
| `idx_tree_parent` | `tree_node` | `(tree_id, parent_id)` | 按 `tree_id` + 父亲快速查直接孩子；分区裁剪 |
| `idx_tree_sibling` | `tree_node` | `(parent_id, sibling_order)` | 取孩子时按 `sibling_order` 顺序遍历 |
| `idx_closure_descendant` | `tree_closure` | `(descendant)` | "从后代看祖先"反向扫描 |
| `idx_closure_depth` | `tree_closure` | `(descendant, depth)` | 反向扫描中按 `depth` 过滤（如取直接父亲 `depth = 1`） |

> 主键索引 `tree_closure(ancestor, descendant)` 已经覆盖了"从祖先看后代"，所以无须再单独建 `(ancestor, ...)` 索引。

#### 完整 DDL

与 `db.py` 中的 `_SCHEMA_SQL` 一字一致：

```sql
-- ⚠ 本 schema 不向后兼容历史版本：tree_node 新增 sibling_order 列、
-- tree_index.root_name 新增 UNIQUE 约束。如有旧 .db 文件请重建。
CREATE TABLE IF NOT EXISTS tree_node (
    id            INTEGER PRIMARY KEY AUTOINCREMENT,
    tree_id       INTEGER NOT NULL,
    parent_id     INTEGER NULL,
    name          TEXT    NOT NULL,
    payload       TEXT,                                       -- JSON 文本
    sibling_order INTEGER NOT NULL DEFAULT 0,                 -- 同父亲下的相对顺序
    node_order    INTEGER NOT NULL DEFAULT 0,                 -- 派生：与 tree.py 的
                                                              -- TreeNode.order 同语义；
                                                              -- 由 _recalc_order(tree_id)
                                                              -- 在所有结构变更后整棵刷新
    created_at    TEXT    NOT NULL DEFAULT (datetime('now')),
    FOREIGN KEY (parent_id) REFERENCES tree_node(id) ON DELETE CASCADE
);

CREATE INDEX IF NOT EXISTS idx_tree_parent
    ON tree_node(tree_id, parent_id);
CREATE INDEX IF NOT EXISTS idx_tree_sibling
    ON tree_node(parent_id, sibling_order);

CREATE TABLE IF NOT EXISTS tree_closure (
    ancestor   INTEGER NOT NULL,
    descendant INTEGER NOT NULL,
    depth      INTEGER NOT NULL CHECK (depth >= 0),
    PRIMARY KEY (ancestor, descendant),
    FOREIGN KEY (ancestor)   REFERENCES tree_node(id) ON DELETE CASCADE,
    FOREIGN KEY (descendant) REFERENCES tree_node(id) ON DELETE CASCADE
);

CREATE INDEX IF NOT EXISTS idx_closure_descendant
    ON tree_closure(descendant);
CREATE INDEX IF NOT EXISTS idx_closure_depth
    ON tree_closure(descendant, depth);

-- tree_index：tree_id ↔ 根节点名 的关联表（人类可读索引）。
-- 由下方触发器全自动维护：新增根、根改名、删根都会同步。
-- root_name UNIQUE：强制"按根名能唯一定位一棵树"；写入端会先做友好检查
-- 再插入，所以这里的 UNIQUE 主要做兜底。
CREATE TABLE IF NOT EXISTS tree_index (
    tree_id    INTEGER PRIMARY KEY,
    root_name  TEXT    NOT NULL UNIQUE,
    updated_at TEXT    NOT NULL DEFAULT (datetime('now'))
);
```

与 2.1 标准 DDL 的对照：

| 维度 | 2.1 标准 DDL（PostgreSQL） | 2.4 本仓库（SQLite） |
| --- | --- | --- |
| 主键类型 | `BIGSERIAL` | `INTEGER PRIMARY KEY AUTOINCREMENT` |
| `payload` 类型 | `JSONB` | `TEXT`（自行 `json.dumps`） |
| 时间戳类型 | `TIMESTAMPTZ DEFAULT now()` | `TEXT DEFAULT (datetime('now'))` |
| 顺序信息 | 无 | `tree_node.sibling_order` + `tree_node.node_order` |
| 反查 `tree_id` | 无 | `tree_index(tree_id, root_name UNIQUE, updated_at)` |
| 索引 | `idx_tree_parent`、`idx_closure_descendant`、`idx_closure_depth` | 同左 + `idx_tree_sibling(parent_id, sibling_order)` |

⚠ **schema 不向后兼容**：旧版本 `tree_node` 没有 `sibling_order` 列，直接复用旧 `.db` 文件会因为 `CREATE INDEX ... ON tree_node(parent_id, sibling_order)` 报 `no such column: sibling_order`。如有旧库请删除重建。

## 3. 增删改查全套 SQL

### 3.1 取整棵子树

```sql
SELECT n.*
FROM tree_node n
JOIN tree_closure c ON c.descendant = n.id
WHERE c.ancestor = :id;
```

包不包括自己？取决于是否要 `c.depth >= 0`（默认包含自己）还是 `c.depth > 0`（不包含自己）。

### 3.2 取祖先链

```sql
SELECT n.*
FROM tree_node n
JOIN tree_closure c ON c.ancestor = n.id
WHERE c.descendant = :id AND c.depth > 0
ORDER BY c.depth DESC;
```

`ORDER BY depth DESC` 让结果是"从根到 `:id` 的父亲"的顺序。

### 3.3 取直接孩子

```sql
SELECT n.*
FROM tree_node n
JOIN tree_closure c ON c.descendant = n.id
WHERE c.ancestor = :id AND c.depth = 1;
```

或者直接用 `parent_id`（更快、不依赖闭包）：

```sql
SELECT * FROM tree_node WHERE parent_id = :id;
```

### 3.4 取某节点的深度

```sql
SELECT MAX(depth) AS depth
FROM tree_closure
WHERE descendant = :id;
```

### 3.5 取某棵子树的最大深度

```sql
SELECT MAX(depth) AS max_depth
FROM tree_closure
WHERE ancestor = :root_id;
```

### 3.6 判断 A 是否是 B 的祖先

```sql
SELECT EXISTS(
    SELECT 1 FROM tree_closure
    WHERE ancestor = :a AND descendant = :b
);
```

这是闭包表最优雅的特性之一：**判断祖先关系 = 一次主键命中**，O(1)。

## 4. 写操作：插入 / 删除 / 移动

闭包表的所有写操作都要"两张表一起改"，强烈建议封装到事务 + 存储过程里。

### 4.1 插入新节点

挂到 `:parent_id` 下的新节点 `:new_id`：

```sql
BEGIN;

-- 1. 插入节点
INSERT INTO tree_node (id, tree_id, parent_id, name, payload)
VALUES (:new_id, :tree_id, :parent_id, :name, :payload);

-- 2. 插入"自反"
INSERT INTO tree_closure (ancestor, descendant, depth)
VALUES (:new_id, :new_id, 0);

-- 3. 把"父亲的所有祖先 + 父亲自己"全部连到 :new_id
INSERT INTO tree_closure (ancestor, descendant, depth)
SELECT c.ancestor, :new_id, c.depth + 1
FROM tree_closure c
WHERE c.descendant = :parent_id;

COMMIT;
```

复杂度：插入需要写 `(深度 + 1)` 行闭包记录，远小于嵌套集的 O(N)。

### 4.2 插入根节点

更简单——只有一条"自反"记录：

```sql
INSERT INTO tree_node (id, tree_id, parent_id, name) VALUES (:id, :tree_id, NULL, :name);
INSERT INTO tree_closure (ancestor, descendant, depth) VALUES (:id, :id, 0);
```

### 4.3 删除整棵子树

如果设了 `ON DELETE CASCADE`：

```sql
DELETE FROM tree_node
WHERE id IN (SELECT descendant FROM tree_closure WHERE ancestor = :id);
```

闭包表里的相关行会被外键级联删干净。

如果想手动控制：

```sql
BEGIN;

-- 删闭包：所有"涉及子树成员"的关系
DELETE FROM tree_closure
WHERE descendant IN (SELECT descendant FROM tree_closure WHERE ancestor = :id);

-- 删节点
DELETE FROM tree_node
WHERE id IN (SELECT id FROM tree_node WHERE id = :id) -- 略，按上面派生
;

COMMIT;
```

### 4.4 仅删除一个节点（保留孩子，挂到祖父）

```sql
BEGIN;

-- 1. 把孩子的 parent_id 上提
UPDATE tree_node SET parent_id = (SELECT parent_id FROM tree_node WHERE id = :id)
WHERE parent_id = :id;

-- 2. 删闭包：把节点 :id 完全摘掉
DELETE FROM tree_closure WHERE ancestor = :id OR descendant = :id;

-- 3. 闭包"绕过":id：把 :id 的祖先们重新连到 :id 的孩子们
INSERT INTO tree_closure (ancestor, descendant, depth)
SELECT a.ancestor, d.descendant, a.depth + d.depth - 1
FROM tree_closure a
JOIN tree_closure d ON TRUE
WHERE a.descendant = :parent_id   -- :id 的祖先们
  AND d.ancestor = :id            -- :id 的子孙们（但 :id 已经删了，要在删之前先取）
  AND d.descendant <> :id;

-- 4. 最后删节点
DELETE FROM tree_node WHERE id = :id;

COMMIT;
```

实现要点：步骤 3 的 JOIN 必须在删闭包之前算好，可以用 CTE 或临时表暂存。

### 4.5 移动子树

把以 `:n` 为根的子树挂到新父亲 `:m` 下。这是闭包表最复杂的写操作，但也比嵌套集简单得多——**只动闭包表里"跨越子树边界"的那些行**。

```sql
BEGIN;

-- 1. 删除"子树外的祖先" → "子树内的所有节点"的旧关系
DELETE FROM tree_closure
WHERE descendant IN (SELECT descendant FROM tree_closure WHERE ancestor = :n)
  AND ancestor   NOT IN (SELECT descendant FROM tree_closure WHERE ancestor = :n);

-- 2. 重新插入"新父亲及其祖先" × "子树内每个节点"的新关系
INSERT INTO tree_closure (ancestor, descendant, depth)
SELECT super.ancestor, sub.descendant, super.depth + sub.depth + 1
FROM tree_closure super       -- :m 及其祖先
JOIN tree_closure sub ON TRUE -- :n 子树内的所有节点
WHERE super.descendant = :m
  AND sub.ancestor = :n;

-- 3. 同步邻接表
UPDATE tree_node SET parent_id = :m WHERE id = :n;

COMMIT;
```

复杂度：`O(子树大小 × 新祖先链长度)`，比嵌套集小一个数量级。

### 4.6 校验环

闭包表自带"防环侦察兵"——只要看新祖先里是否包含自己：

```sql
SELECT EXISTS(
    SELECT 1 FROM tree_closure
    WHERE ancestor = :n AND descendant = :m
);
-- 如果返回 true，说明 :m 在 :n 的子树里，移动会形成环
```

## 5. 性能分析

| 操作 | 复杂度 | 说明 |
| --- | --- | --- |
| 取子树 | O(子树大小) | 一次主键 / 索引扫描 |
| 取祖先 | O(深度) | 一次索引扫描 |
| 取直接孩子 | O(孩子数) | 用 `depth = 1` 或 `parent_id` |
| 判 A 是否 B 的祖先 | O(1) | 主键命中 |
| 插入节点 | O(深度) | 写 `深度 + 1` 行闭包 |
| 删除节点 | O(子树大小) | 闭包级联 |
| 移动子树 | O(子树大小 × 新深度) | 拆 / 重连闭包 |
| 空间 | O(N × 平均深度) | 是闭包表唯一明显的额外成本 |

**关键结论**：闭包表用"一份独立的派生索引 + 适量空间"换来了"几乎所有读操作都是一次 JOIN"的均衡。

## 6. 一致性维护

闭包表的最大风险不是性能，而是"闭包表与节点表不一致"。两条防线：

### 6.1 应用层封装

把"插入 / 删除 / 移动"封装成 3~4 个 helper 函数（`add_node`、`remove_node`、`move_subtree`、`graft_subtree`），业务代码只调用这些 helper，禁止直接操作 `tree_closure`。

### 6.2 数据库触发器（可选但强烈推荐）

PostgreSQL 例子：

```sql
CREATE OR REPLACE FUNCTION tree_node_after_insert() RETURNS TRIGGER AS $$
BEGIN
    INSERT INTO tree_closure (ancestor, descendant, depth) VALUES (NEW.id, NEW.id, 0);
    IF NEW.parent_id IS NOT NULL THEN
        INSERT INTO tree_closure (ancestor, descendant, depth)
        SELECT c.ancestor, NEW.id, c.depth + 1
        FROM tree_closure c
        WHERE c.descendant = NEW.parent_id;
    END IF;
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_tree_node_after_insert
AFTER INSERT ON tree_node
FOR EACH ROW EXECUTE FUNCTION tree_node_after_insert();
```

类似地写 `AFTER UPDATE` 处理 `parent_id` 变更（移动子树）、`AFTER DELETE` 不需要写——靠外键 CASCADE 即可。

有了触发器，业务代码只动 `tree_node`，闭包表自动维护。这是最稳妥的方案。

#### 6.2.1 本仓库的 SQLite 触发器实现

`db.py` 里 `_TRIGGERS_SQL` 一次创建 6 个触发器，按职责分两组：

**A. 闭包表自动维护**（对应文档 4.1 / 4.5 节）：

| # | 触发器 | 时机 | 作用 |
| --- | --- | --- | --- |
| 1 | `trg_tree_node_after_insert` | `AFTER INSERT ON tree_node` | 写入自反 `(NEW.id, NEW.id, 0)`，并把"父亲所有祖先"链接到新节点；插入根（`parent_id IS NULL`）时第二条 SELECT 自然为空 |
| 2 | `trg_tree_node_after_move` | `AFTER UPDATE OF parent_id` | 移动子树时按 4.5 节算法，先删"子树外祖先 → 子树内任意节点"的旧关系，再插"新祖先链 × 子树内所有节点"的新关系 |

`AFTER DELETE` **故意不写**——`tree_closure` 的两个外键都带 `ON DELETE CASCADE`，删节点时闭包行被数据库自动级联清理。

**B. `tree_index` 自动维护**（让"按根节点名反查 tree_id"始终是真值）：

| # | 触发器 | 时机 | 作用 |
| --- | --- | --- | --- |
| 3 | `trg_tree_index_after_root_insert` | `AFTER INSERT`，`WHEN NEW.parent_id IS NULL` | 新增根 → `UPSERT tree_index`，让 `save_tree` 这种"先 DELETE 再 INSERT"的覆盖式写入也能正确刷新 `root_name` |
| 4 | `trg_tree_index_after_root_rename` | `AFTER UPDATE OF name`，`WHEN NEW.parent_id IS NULL` | 根改名 → 同步索引行的 `root_name` |
| 5 | `trg_tree_index_after_root_delete` | `AFTER DELETE`，`WHEN OLD.parent_id IS NULL` | 删根 → 若该 `tree_id` 还有别的根则 UPSERT 指向"剩余编号最小的根"，否则删掉索引行；非根节点删除被 `WHEN` 拦掉，所以 `ON DELETE CASCADE` 带下来的子孙不会反复触发，性能 `O(根数量)` 而非 `O(节点数)` |
| 6 | `trg_tree_index_after_parent_change` | `AFTER UPDATE OF parent_id`，`WHEN` 节点"是否为根"的状态发生翻转 | 覆盖两种情形：非根→根（提升）UPSERT 索引行；根→非根（降级）若还有别的根则 UPSERT，否则删行 |

完整 DDL（与 `db.py` 中的 `_TRIGGERS_SQL` 一字一致）：

```sql
-- 1) 插入节点：写入自反记录 + 把父亲所有祖先连到新节点
CREATE TRIGGER IF NOT EXISTS trg_tree_node_after_insert
AFTER INSERT ON tree_node
FOR EACH ROW
BEGIN
    INSERT INTO tree_closure(ancestor, descendant, depth)
        VALUES (NEW.id, NEW.id, 0);

    INSERT INTO tree_closure(ancestor, descendant, depth)
        SELECT c.ancestor, NEW.id, c.depth + 1
        FROM   tree_closure c
        WHERE  c.descendant = NEW.parent_id;
END;

-- 2) 移动子树：UPDATE parent_id 时整段拆旧 + 重连
--    实质就是文档 4.5 节的 SQL，只是把 :n / :m 换成 NEW.id / NEW.parent_id。
CREATE TRIGGER IF NOT EXISTS trg_tree_node_after_move
AFTER UPDATE OF parent_id ON tree_node
FOR EACH ROW
WHEN OLD.parent_id IS NOT NEW.parent_id
BEGIN
    -- 2.1 删"子树外祖先 → 子树内任意节点"的旧关系
    DELETE FROM tree_closure
    WHERE descendant IN (
        SELECT descendant FROM tree_closure WHERE ancestor = NEW.id
    )
    AND ancestor NOT IN (
        SELECT descendant FROM tree_closure WHERE ancestor = NEW.id
    );

    -- 2.2 写"新祖先链 × 子树内所有节点"的新关系
    --     这里语义就是笛卡尔积，写成显式 CROSS JOIN 比"JOIN 不带 ON"更清晰。
    INSERT INTO tree_closure(ancestor, descendant, depth)
        SELECT s.ancestor,
               d.descendant,
               s.depth + d.depth + 1
        FROM       tree_closure s
        CROSS JOIN tree_closure d
        WHERE  s.descendant = NEW.parent_id
          AND  d.ancestor   = NEW.id;
END;

-- 3) tree_index 维护：新增根节点 → 写入 / 覆盖索引行。
--    parent_id IS NULL 的节点即根节点；用 UPSERT 让 save_tree 这种
--    "先 DELETE 再 INSERT" 的覆盖式写入也能正确刷新 root_name。
CREATE TRIGGER IF NOT EXISTS trg_tree_index_after_root_insert
AFTER INSERT ON tree_node
FOR EACH ROW
WHEN NEW.parent_id IS NULL
BEGIN
    INSERT INTO tree_index(tree_id, root_name)
        VALUES (NEW.tree_id, NEW.name)
    ON CONFLICT(tree_id) DO UPDATE
        SET root_name  = excluded.root_name,
            updated_at = datetime('now');
END;

-- 4) tree_index 维护：根节点改名 → 同步索引行。
CREATE TRIGGER IF NOT EXISTS trg_tree_index_after_root_rename
AFTER UPDATE OF name ON tree_node
FOR EACH ROW
WHEN NEW.parent_id IS NULL
BEGIN
    UPDATE tree_index
       SET root_name  = NEW.name,
           updated_at = datetime('now')
     WHERE tree_id = NEW.tree_id;
END;

-- 5) tree_index 维护：根节点被删 → 索引行回退到"剩余编号最小的根"，
--    若该 tree_id 下已无任何根则删除索引行。
--    非根节点删除会被 WHEN 拦掉，所以 ON DELETE CASCADE 把子孙带下来时
--    不会触发这条逻辑，性能上等于 O(根数量) 而非 O(节点总数)。
CREATE TRIGGER IF NOT EXISTS trg_tree_index_after_root_delete
AFTER DELETE ON tree_node
FOR EACH ROW
WHEN OLD.parent_id IS NULL
BEGIN
    -- 还有别的根：tree_index 改指向编号最小的剩余根。
    UPDATE tree_index
       SET root_name  = (
               SELECT name FROM tree_node
               WHERE tree_id = OLD.tree_id AND parent_id IS NULL
               ORDER BY id LIMIT 1
           ),
           updated_at = datetime('now')
     WHERE tree_id = OLD.tree_id
       AND EXISTS (
           SELECT 1 FROM tree_node
           WHERE tree_id = OLD.tree_id AND parent_id IS NULL
       );
    -- 没有任何根：清掉索引行。
    DELETE FROM tree_index
     WHERE tree_id = OLD.tree_id
       AND NOT EXISTS (
           SELECT 1 FROM tree_node
           WHERE tree_id = OLD.tree_id AND parent_id IS NULL
       );
END;

-- 6) tree_index 维护：UPDATE parent_id 改变了节点"是否为根"时同步索引。
--    覆盖两种情形：
--      a) 非根 → 根（提升）：UPSERT 索引行指向编号最小的根
--      b) 根 → 非根（降级）：若还有其他根 UPSERT 指向编号最小的，否则删行
--    ⚠ 配合代码层"一棵 tree_id 一个根"的不变量使用：写入端会拒绝
--    重复根，所以"提升时已有别的根"这种脏状态实际进不来。
CREATE TRIGGER IF NOT EXISTS trg_tree_index_after_parent_change
AFTER UPDATE OF parent_id ON tree_node
FOR EACH ROW
WHEN (OLD.parent_id IS NULL AND NEW.parent_id IS NOT NULL)
  OR (OLD.parent_id IS NOT NULL AND NEW.parent_id IS NULL)
BEGIN
    INSERT INTO tree_index(tree_id, root_name)
        SELECT NEW.tree_id, name
        FROM   tree_node
        WHERE  tree_id = NEW.tree_id AND parent_id IS NULL
        ORDER BY id LIMIT 1
    ON CONFLICT(tree_id) DO UPDATE
        SET root_name  = excluded.root_name,
            updated_at = datetime('now');
    DELETE FROM tree_index
     WHERE tree_id = NEW.tree_id
       AND NOT EXISTS (
           SELECT 1 FROM tree_node
           WHERE tree_id = NEW.tree_id AND parent_id IS NULL
       );
END;
```

有了这 6 个触发器，业务代码 **只需要操作 `tree_node`**，`tree_closure` 和 `tree_index` 都是自动派生出来的——这正是 6.2 节"最稳妥的方案"的实际落地。

### 6.3 离线对账

定期跑一段 SQL 比对"递归 CTE 算出的闭包"和"实际闭包表"是否一致，发现漂移立即报警 + 修复。

## 7. 典型陷阱

### 7.1 没有自反记录

很多新手实现闭包表时漏掉 `(self, self, 0)`，导致"取子树时是否包含自己"的逻辑分散到各处。**永远从一开始就插入自反记录**，统一所有边界。

### 7.2 闭包表膨胀

闭包表行数 = `O(N × 平均深度)`。对超深 + 大量节点的树（例如百万节点 + 平均深度 50），闭包表会膨胀到 5000 万行。应对：

- 拆分多棵树，按 `tree_id` 物理分区。
- 限制最大深度（业务上几乎都有上限）。
- 必要时改用方案二的 `ltree`（空间更小）。

### 7.3 移动子树写错 SQL

第 4.5 节的 SQL 一旦写错（条件挂错位置）会留下"幽灵关系"。**务必用单元测试 + 离线对账**双重保护。

### 7.4 没有用事务

闭包表所有写操作都要原子地修改两张表，**裸跑 SQL 会留下不一致**。最低限度：所有写操作都在同一个事务里。

### 7.5 用 `descendant` 索引代替主键时的反例

只建 `(ancestor, descendant)` 主键 + `(descendant)` 单列索引，不要建 `(descendant, ancestor)` 单独索引——`descendant` + 主键已经能加速所有"反向"查询。

---

## 8. 优缺点

### 8.1 优点

- **所有读操作都极快**：取子树 / 取祖先 / 判祖先关系全部一次 JOIN。
- **写代价中等且可预测**：插入只动 `深度 + 1` 行。
- **强一致**：可以用外键 + 触发器在数据库层面保证。
- **无深度限制**。
- **判断祖先关系 = O(1) 主键命中**：是其它方案羡慕不来的能力。
- **支持 DAG 的最小改动**：把闭包表里的"祖先-后代"放宽成"指向-被指向"就行。

### 8.2 缺点

- **空间开销**：闭包表行数 `O(N × 深度)`。
- **写需要操作两张表**：必须封装好或用触发器。
- **移动子树的 SQL 比邻接表复杂**：但比嵌套集简单得多。

---

## 9. 适用场景

闭包表的"通用最优"特性使它适合大多数中大型业务：

1. **组织架构 / 权限继承**：经常要"判某用户是否在某部门子树下"。
2. **知识库 / Wiki / CMS 多级目录**：每个页面都要展示面包屑。
3. **类目 / 分类树**：经常要按类目子树聚合销量、库存等指标。
4. **任何要支持 DAG（多父亲）的场景**：闭包表天然支持，只是空间会进一步膨胀。
5. **多租户 SaaS**：每个租户一棵树，靠 `tree_id` 分区，闭包表也按租户分区即可。

不太适合的场景：

- **节点数极多（亿级）+ 深度也大**的树：闭包表空间爆炸。
- **几乎只读 + 极少 join**：用方案二（物化路径）会更省。

## 10. 与 `tree.py` 的映射

| 内存模型 | 闭包表 |
| --- | --- |
| `TreeNode.value` | `tree_node.payload` |
| `TreeNode.name` | `tree_node.name` |
| `TreeNode.parent` | `tree_node.parent_id` |
| `TreeNode.children` | `WHERE parent_id = self.id` |
| `TreeNode.level` | `SELECT MAX(depth) FROM tree_closure WHERE descendant = self.id` 或冗余列 |
| `TreeNode.order` | 需要冗余列 `dfs_order`，由 `Tree.refresh_orders` 之后回写 |
| `Tree.root` | `parent_id IS NULL` 或闭包表里 `ancestor = descendant AND ancestor 不出现在其它行的 descendant 中` |

### 10.1 把 `tree.py` 树落到闭包表

```python
def save_to_closure(conn, tree_id: int, tree: Tree) -> None:
    if tree.root is None:
        return

    # 1. 先用 BFS 写节点表，记录每个节点的新 id
    id_map: dict[int, int] = {}
    queue = deque([(tree.root, None)])
    while queue:
        node, parent_db_id = queue.popleft()
        new_id = conn.execute(
            "INSERT INTO tree_node (tree_id, parent_id, name, payload) "
            "VALUES (%s, %s, %s, %s) RETURNING id",
            (tree_id, parent_db_id, node.name, node.value),
        ).fetchone().id
        id_map[id(node)] = new_id
        for child in node.children:
            queue.append((child, new_id))

    # 2. 写闭包表（自反 + 祖先链）
    for node in tree.bfs():
        my_id = id_map[id(node)]
        conn.execute(
            "INSERT INTO tree_closure (ancestor, descendant, depth) VALUES (%s, %s, 0)",
            (my_id, my_id),
        )
        ancestor = node.parent
        depth = 1
        while ancestor is not None:
            conn.execute(
                "INSERT INTO tree_closure (ancestor, descendant, depth) VALUES (%s, %s, %s)",
                (id_map[id(ancestor)], my_id, depth),
            )
            ancestor = ancestor.parent
            depth += 1
```

如果设置了触发器，那么第 2 步可以省掉——触发器会在第 1 步插入 `tree_node` 时自动维护闭包。

### 10.2 反向加载（极简版）

只读节点表 + 重建对象引用，闭包表不参与"重建"，仅用于查询加速：

```python
def load_from_closure(conn, tree_id: int) -> Tree:
    rows = conn.execute(
        "SELECT id, parent_id, name, payload FROM tree_node WHERE tree_id = %s",
        (tree_id,),
    ).fetchall()

    nodes = {r.id: TreeNode(value=r.payload, name=r.name) for r in rows}
    root = None
    for r in rows:
        node = nodes[r.id]
        if r.parent_id is None:
            root = node
        else:
            parent = nodes[r.parent_id]
            node.parent = parent
            parent.children.append(node)

    tree = Tree()
    tree.root = root
    if root is not None:
        tree.refresh_levels()
        tree.refresh_orders()
    return tree
```

注意：闭包表不"重建" `tree.py` 内存模型，它只在数据库侧给"查祖先 / 子树"提速。**`tree.py` 内存模型对应的真值始终是 `parent_id`。**

## 11. 一句话总结

> **闭包表 = 邻接表 + 一份"祖先-后代"派生索引。** 它把"读快"和"写快"做到了同时兼顾，并且能用外键 + 触发器在数据库层强一致地维护。如果你不知道选哪个，就选它。
