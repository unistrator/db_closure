# 方案一：邻接表（Adjacency List）

> 树的最朴素存储方式：**每行一个节点，外加一列 `parent_id` 指向父节点**。它是大部分项目的默认起点，也是后面几种方案的"对照基准"。

---

## 1. 核心思想

把一棵树拆成"节点"和"父子关系"两件事，但**不单独建一张关系表**——直接把"父亲是谁"当成节点行的一个外键列存进去。

- 表中每一行 = 一个 `TreeNode`
- `parent_id` 列 = 这个节点指向父亲的"指针"
- `parent_id IS NULL` = 这个节点是根
- 一张表里可以同时存任意多棵树（每棵树有自己的根）

整张表本质上就是一张**有向边表**：`(child_id → parent_id)`。

---

## 2. 表结构

### 2.1 PostgreSQL / MySQL 8+ 通用 DDL

```sql
CREATE TABLE tree_node (
    id          BIGSERIAL    PRIMARY KEY,
    parent_id   BIGINT       NULL,
    tree_id     BIGINT       NOT NULL,            -- 多棵树共存时的"分组键"
    name        VARCHAR(255) NOT NULL,
    payload     JSONB,                            -- 业务字段统一塞进来
    created_at  TIMESTAMPTZ  NOT NULL DEFAULT now(),
    updated_at  TIMESTAMPTZ  NOT NULL DEFAULT now(),

    CONSTRAINT fk_parent
        FOREIGN KEY (parent_id) REFERENCES tree_node(id) ON DELETE CASCADE
);

CREATE INDEX idx_tree_parent ON tree_node (tree_id, parent_id);
CREATE INDEX idx_tree_root   ON tree_node (tree_id) WHERE parent_id IS NULL;
```

### 2.2 关键约束与索引

- **`parent_id` 自引用外键**：让数据库帮你保证"父亲必须存在"，再加 `ON DELETE CASCADE`，删根时自动清掉整棵树。
- **`(tree_id, parent_id)` 复合索引**：取某棵树某个节点的孩子时这是热路径。
- **部分索引 `WHERE parent_id IS NULL`**：根节点只占少数，单独索引可以让"列出所有根 / 所有树"非常快。
- **可选：`UNIQUE (parent_id, name)`**：避免同一父亲下有两个同名子节点（菜单 / 目录场景常用）。
- **可选：`CHECK (id <> parent_id)`**：在 DDL 层面拒绝"自己当自己父亲"的脏数据。

### 2.3 防环约束

邻接表本身**无法在 DDL 层面**完全防止环（`A → B → C → A`）。可选做法：

1. **应用层**：每次写 `parent_id` 之前先判断"新父亲是否在自己子树里"。
2. **触发器**：写入时跑一段递归 CTE 检查。
3. **闭包表辅助**：见 [方案四](./方案04-闭包表.md)。

## 3. 增删改查全套 SQL

### 3.1 插入根节点

```sql
INSERT INTO tree_node (tree_id, parent_id, name, payload)
VALUES (:tree_id, NULL, :name, :payload)
RETURNING id;
```

### 3.2 在某节点下挂新子节点

```sql
INSERT INTO tree_node (tree_id, parent_id, name, payload)
VALUES (:tree_id, :parent_id, :name, :payload)
RETURNING id;
```

### 3.3 取直接孩子（一层）

```sql
SELECT * FROM tree_node
WHERE parent_id = :id
ORDER BY id;
```

### 3.4 取祖先链（无 CTE 时）

```sql
-- 应用层伪代码
node = SELECT * FROM tree_node WHERE id = :id;
while node.parent_id IS NOT NULL:
    node = SELECT * FROM tree_node WHERE id = node.parent_id;
    yield node;
```

每条祖先就是一次往返查询，**深度即查询次数**——这是邻接表最大的痛点。

### 3.5 取祖先链（递归 CTE）

```sql
WITH RECURSIVE ancestors AS (
    SELECT * FROM tree_node WHERE id = :id
    UNION ALL
    SELECT p.*
    FROM tree_node p
    JOIN ancestors a ON p.id = a.parent_id
)
SELECT * FROM ancestors WHERE id <> :id;
```

### 3.6 取整棵子树（递归 CTE）

```sql
WITH RECURSIVE subtree AS (
    SELECT *, 0 AS depth FROM tree_node WHERE id = :root_id
    UNION ALL
    SELECT n.*, s.depth + 1
    FROM tree_node n
    JOIN subtree s ON n.parent_id = s.id
)
SELECT * FROM subtree
ORDER BY depth, id;
```

### 3.7 移动子树

把 `n` 整体挂到新父亲 `m` 下：**只改一行**。

```sql
UPDATE tree_node SET parent_id = :m_id WHERE id = :n_id;
```

但要注意：

- 必须先校验"`m` 不在 `n` 的子树里"，否则会形成环。
- 校验本身要走一次递归 CTE（或借助闭包表）。

### 3.8 删除节点

```sql
-- 删除节点本身 + 整棵子树（依赖外键 ON DELETE CASCADE）
DELETE FROM tree_node WHERE id = :id;

-- 仅删除节点本身、保留孩子（把孩子挂到祖父）
UPDATE tree_node SET parent_id = (SELECT parent_id FROM tree_node WHERE id = :id)
WHERE parent_id = :id;
DELETE FROM tree_node WHERE id = :id;
```

## 4. 性能分析

| 操作 | 不用 CTE | 用递归 CTE |
| --- | --- | --- |
| 插入节点 | O(1) | O(1) |
| 移动子树 | O(1) | O(1)（+ 一次环检查 O(深度)） |
| 删除子树 | O(子树大小)（CASCADE） | 同 |
| 取直接孩子 | O(孩子数) | 同 |
| 取祖先链 | O(深度) **次往返** | O(深度) 单次查询 |
| 取整棵子树 | 不可一次完成 | O(子树大小) 单次查询 |

**关键结论**：邻接表的写入永远是 O(1)，但跨层级读取的成本和"深度"成正比，并且严重依赖 CTE。如果你的数据库不支持递归 CTE（或非常老的 MySQL 5.x），整棵子树查询会非常痛苦。

---

## 5. 典型陷阱

### 5.1 N+1 查询

最常见的反模式：在应用层用 ORM 一边遍历一边触发查询。

```python
# 反例：每个 child.parent 都会触发一次 SQL
for child in children:
    print(child.parent.name)
```

解决方案：

- 一次 CTE 把整棵子树拉回来，在内存里组装。
- 或者改用闭包表 / 物化路径。

### 5.2 删大子树时锁表

`DELETE ... CASCADE` 会沿着外键一层层删，子树越大锁持有时间越长。生产环境删大子树建议：

1. 用 CTE 拿到所有 id；
2. 分批 `DELETE FROM tree_node WHERE id = ANY(:batch)`。

### 5.3 没有 `tree_id` 的多租户灾难

很多项目一开始没考虑"多棵树"，后期发现要按租户 / 项目隔离时不得不大改表。**新表一律带上 `tree_id`**（哪怕暂时只有一棵树）。

### 5.4 递归 CTE 的方言差异

| 数据库 | 语法要点 |
| --- | --- |
| PostgreSQL | `WITH RECURSIVE`，支持 `CYCLE` 子句防环 |
| MySQL 8+ | `WITH RECURSIVE`，需手动处理深度限制 `cte_max_recursion_depth` |
| SQL Server | `WITH`（无需 `RECURSIVE` 关键字）+ `OPTION (MAXRECURSION N)` |
| Oracle | 推荐 `CONNECT BY ... PRIOR`，更老更顺手 |
| SQLite | 3.8.3+ 支持 `WITH RECURSIVE` |

## 6. 优缺点

### 6.1 优点

- **概念极简**：一列搞定，新人 5 分钟看懂。
- **写入极快**：所有结构修改都是 O(1)。
- **强一致**：靠数据库的外键 + 事务即可。
- **空间最小**：除了 `parent_id` 没有任何冗余。

### 6.2 缺点

- **跨层级读慢**：祖先链 / 子树都依赖递归 CTE 或多次往返。
- **环检测难**：DDL 层防不住，只能靠应用 / 触发器。
- **聚合查询费劲**：例如"某节点子树下所有叶子的总销量"必须先递归再聚合。

---

## 7. 适用场景

直接用邻接表的最佳场景：

1. **节点总量不大**（< 几十万）。
2. **树的深度不深**（< 10 层）。
3. **跨层级查询不在热路径上**：用户大部分时间只关心"上下一级"。
4. **写多读少**或写读比相当。

典型业务：

- 后台菜单树
- 部门 / 岗位组织结构
- 评论的两三级回复
- 动态表单 / 配置树
- 文件夹（深度浅时）

不适合直接用邻接表的场景：

- 商品类目树需要频繁取整棵子树聚合销量
- 知识库 / Wiki 经常查"某页面的所有祖先做面包屑"且每页都要展示
- 权限系统需要判断"用户是否在某分组的子树里"且 QPS 很高

这些场景应该叠加 [方案二（物化路径）](./方案02-路径枚举.md) 或升级到 [方案四（闭包表）](./方案04-闭包表.md)。

## 8. 与 `tree.py` 的映射

| 内存模型 | 邻接表列 |
| --- | --- |
| `TreeNode.value` | `payload`（或单独一列） |
| `TreeNode.name` | `name` |
| `TreeNode.parent` | `parent_id` |
| `TreeNode.children` | 不存，由 `WHERE parent_id = self.id` 派生 |
| `TreeNode.level` | 可冗余一列 `depth`，写入时维护 |
| `TreeNode.order` | 可冗余一列 `dfs_order`，由 `Tree.refresh_orders` 之后批量回写 |
| `Tree.root` | `parent_id IS NULL AND tree_id = :id` |

### 8.1 加载到内存的最小代码

```python
def load_tree(conn, tree_id: int) -> Tree:
    rows = conn.execute(
        "SELECT id, parent_id, name, payload FROM tree_node WHERE tree_id = %s",
        (tree_id,),
    ).fetchall()

    nodes: dict[int, TreeNode] = {
        r.id: TreeNode(value=r.payload, name=r.name) for r in rows
    }
    root: TreeNode | None = None
    for r in rows:
        node = nodes[r.id]
        if r.parent_id is None:
            root = node
        else:
            parent = nodes[r.parent_id]
            node.parent = parent
            parent.children.append(node)

    tree = Tree()
    tree.root = root
    if root is not None:
        tree.refresh_levels()
        tree.refresh_orders()
    return tree
```

### 8.2 写回数据库的最小代码

```python
def save_tree(conn, tree_id: int, tree: Tree) -> None:
    conn.execute("DELETE FROM tree_node WHERE tree_id = %s", (tree_id,))
    if tree.root is None:
        return

    # BFS 写入，先父后子，保证 parent_id 的外键已存在
    queue = deque([(tree.root, None)])
    while queue:
        node, parent_id = queue.popleft()
        new_id = conn.execute(
            "INSERT INTO tree_node (tree_id, parent_id, name, payload) "
            "VALUES (%s, %s, %s, %s) RETURNING id",
            (tree_id, parent_id, node.name, node.value),
        ).fetchone().id
        for child in node.children:
            queue.append((child, new_id))
```

注意：上面是"全量重写"的简化实现。生产环境一般用**增量写**——按 `id` 对齐，只 UPDATE 改动过的行，避免每次都炸表。

---

## 9. 一句话总结

> **邻接表是"默认起手式"：一列 `parent_id` 就够了，复杂查询全部交给递归 CTE。** 等到写入压力没问题但读子树 / 读祖先成为瓶颈时，再考虑给它叠一层闭包表或物化路径。
