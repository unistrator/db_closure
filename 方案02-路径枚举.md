# 方案二：路径枚举 / 物化路径（Materialized Path）

> 在每行额外存一个**从根到自己的路径**（字符串或数组），把"祖先关系"直接编码到值里。子树查询退化为一次"前缀匹配"。

---

## 1. 核心思想

如果说邻接表是"指着我父亲的牌子"，那物化路径就是**让每个节点把自己从根到自己的全程路径都贴在身上**。

- 节点 `5` 在 `1 → 4 → 9 → 5` 这条链上 → 它的 `path` 就是 `'/1/4/9/5/'`。
- 想查节点 `9` 的所有后代？一句 `path LIKE '/1/4/9/%'`。
- 想查节点 `5` 的所有祖先？把 `path` 拆开就是 `[1, 4, 9]`，再 `IN` 一下即可。

代价是：**一次"移动子树"会改动子树里所有节点的 `path`。**

物化路径有三种常见的"承载形式"：

| 形式 | 例子 | 适用 |
| --- | --- | --- |
| 字符串路径 | `'/1/4/9/'` | 所有数据库通用 |
| 数组路径 | `ARRAY[1, 4, 9]` | PostgreSQL 等支持 ARRAY 的数据库 |
| `ltree` 类型 | `'root.dept.eng.team_a'` | PostgreSQL 专属（最佳） |

---

## 2. 表结构

### 2.1 字符串路径（通用版）

```sql
CREATE TABLE tree_node (
    id          BIGSERIAL    PRIMARY KEY,
    tree_id     BIGINT       NOT NULL,
    parent_id   BIGINT       NULL,                -- 真值来源，配合 path 一起用
    path        VARCHAR(1024) NOT NULL,            -- '/1/4/9/'
    depth       INT          NOT NULL,            -- 冗余，但查询时极方便
    name        VARCHAR(255) NOT NULL,
    payload     JSONB,

    CONSTRAINT fk_parent FOREIGN KEY (parent_id) REFERENCES tree_node(id) ON DELETE CASCADE
);

CREATE INDEX idx_tree_path  ON tree_node (tree_id, path);
CREATE INDEX idx_tree_depth ON tree_node (tree_id, depth);
```

约定（必须严格遵守，否则后面所有查询都会崩）：

1. `path` **以 `/` 开头并以 `/` 结尾**（前后封口，避免 `'/12/' LIKE '/1/%'` 这种误判）。
2. `path` 末段 = 自己的 `id`。
3. `path` 长度 = `depth + 1`（按 `/` 分段计算）。
4. **`parent_id` + `path` 组成真值的两份副本**，应用层每次写入都要同时更新，最好用触发器或 ORM 钩子统一封装。

### 2.2 数组路径（PostgreSQL）

```sql
CREATE TABLE tree_node (
    id          BIGSERIAL PRIMARY KEY,
    tree_id     BIGINT NOT NULL,
    parent_id   BIGINT NULL REFERENCES tree_node(id) ON DELETE CASCADE,
    ancestors   BIGINT[] NOT NULL,    -- ARRAY[1, 4, 9]
    name        TEXT NOT NULL
);

-- GIN 索引让 @> / <@ / && 类操作走索引
CREATE INDEX idx_tree_ancestors ON tree_node USING GIN (ancestors);
```

数组路径比字符串路径更"干净"：

- 不用关心分隔符；
- 类型安全（ID 就是 BIGINT，不会和字符串混）；
- 用 GIN 索引可以快速做"包含 / 被包含"判断。

### 2.3 PostgreSQL `ltree`（强烈推荐）

```sql
CREATE EXTENSION IF NOT EXISTS ltree;

CREATE TABLE tree_node (
    id   BIGSERIAL PRIMARY KEY,
    path LTREE NOT NULL,                -- 'root.dept.eng.team_a'
    name TEXT NOT NULL
);

CREATE INDEX idx_node_path_gist ON tree_node USING GIST (path);
CREATE INDEX idx_node_path_btree ON tree_node USING BTREE (path);
```

`ltree` 是 PostgreSQL 官方扩展，提供：

- **专用类型**：路径段是字母 / 数字 / 下划线组成的"标签"，`.` 分隔。
- **专用索引**：GiST 或 SP-GiST 可以同时加速 `@>` / `<@` / `~` / `?` 等操作。
- **专用操作符**：见下表。

| 操作符 | 含义 |
| --- | --- |
| `A @> B` | A 是 B 的祖先（A 包含 B） |
| `A <@ B` | A 是 B 的后代 |
| `A ~ lquery` | path 是否匹配某个 lquery 模式（类似 glob） |
| `A ? ltxtquery` | 是否匹配某个全文层级表达式 |

## 3. 增删改查全套 SQL

### 3.1 插入根节点

字符串版：

```sql
INSERT INTO tree_node (tree_id, parent_id, path, depth, name)
VALUES (:tree_id, NULL, '/__placeholder__/', 0, :name)
RETURNING id;

-- 拿到 id 之后回填 path（也可以用触发器自动做）
UPDATE tree_node
SET path = '/' || id || '/'
WHERE id = :new_id;
```

`ltree` 版：

```sql
INSERT INTO tree_node (path, name) VALUES (text2ltree(:label), :name);
```

### 3.2 在某节点下挂新子节点

字符串版（先取父亲的 `path` / `depth`，再拼接）：

```sql
INSERT INTO tree_node (tree_id, parent_id, path, depth, name)
SELECT :tree_id, p.id, p.path || nextval('tree_node_id_seq') || '/', p.depth + 1, :name
FROM tree_node p
WHERE p.id = :parent_id
RETURNING id;
```

更稳的写法是先 `INSERT` 拿到 id，再 `UPDATE` 写 path，避免 `nextval` 与实际 `id` 不一致。

`ltree` 版：

```sql
INSERT INTO tree_node (path, name)
SELECT p.path || :label::ltree, :name
FROM tree_node p
WHERE p.id = :parent_id;
```

### 3.3 取整棵子树

字符串版（前缀扫描，走 `idx_tree_path`）：

```sql
SELECT * FROM tree_node
WHERE tree_id = :tree_id
  AND path LIKE :root_path || '%';
```

数组版：

```sql
SELECT * FROM tree_node
WHERE tree_id = :tree_id
  AND ancestors @> ARRAY[:root_id]::BIGINT[];
```

`ltree` 版：

```sql
SELECT * FROM tree_node WHERE path <@ :root_path;
```

### 3.4 取祖先链

字符串版（先拆 path 再 IN）：

```python
# 应用层
ancestors_ids = [int(s) for s in node.path.strip("/").split("/")[:-1]]
```

```sql
SELECT * FROM tree_node WHERE id = ANY(:ancestors_ids) ORDER BY depth;
```

数组版（一句话搞定）：

```sql
SELECT * FROM tree_node
WHERE id = ANY(:node_ancestors_array)
ORDER BY array_position(:node_ancestors_array, id);
```

`ltree` 版：

```sql
SELECT * FROM tree_node WHERE path @> :node_path AND path <> :node_path
ORDER BY nlevel(path);
```

### 3.5 取直接孩子

字符串版需要"恰好深一层"的过滤：

```sql
SELECT * FROM tree_node
WHERE tree_id = :tree_id
  AND path LIKE :parent_path || '%'
  AND depth = :parent_depth + 1;
```

或者直接走保留下来的 `parent_id`：

```sql
SELECT * FROM tree_node WHERE parent_id = :parent_id;
```

`ltree` 版（`nlevel` 是 `ltree` 内置的"层数"函数）：

```sql
SELECT * FROM tree_node
WHERE path <@ :parent_path AND nlevel(path) = nlevel(:parent_path) + 1;
```

### 3.6 取某个深度的所有节点

```sql
-- 字符串 / 数组版
SELECT * FROM tree_node WHERE tree_id = :tree_id AND depth = :d;

-- ltree 版
SELECT * FROM tree_node WHERE nlevel(path) = :d;
```

## 4. 复杂操作：移动 / 删除子树

### 4.1 移动子树（最关键、最容易出错）

把以 `n` 为根的子树挂到新父亲 `m` 下：

字符串版：

```sql
WITH old AS (SELECT path, depth FROM tree_node WHERE id = :n_id),
     new AS (SELECT path, depth FROM tree_node WHERE id = :m_id)
UPDATE tree_node SET
    path  = (SELECT new.path FROM new) || SUBSTRING(path FROM LENGTH((SELECT old.path FROM old)) - LENGTH('/' || :n_id || '/') + 1),
    depth = depth + (SELECT new.depth + 1 - old.depth FROM old, new),
    parent_id = CASE WHEN id = :n_id THEN :m_id ELSE parent_id END
WHERE path LIKE (SELECT old.path FROM old) || '%';
```

`ltree` 版（极度优雅）：

```sql
UPDATE tree_node
SET path = :new_parent_path || subpath(path, nlevel(:old_subtree_path) - 1)
WHERE path <@ :old_subtree_path;
```

注意事项：

1. **必须事务包裹**：移动期间 `path` 处于"半新半旧"的状态，任何并发读都会读到不一致结果。
2. **必须校验环**：`m` 不能在 `n` 的子树里，否则会构造出"自己包含自己"的诡异结构。
3. **批量更新代价 = 子树大小**：移动 1000 个节点的子树就是 UPDATE 1000 行。

### 4.2 删除子树

```sql
-- 字符串版
DELETE FROM tree_node
WHERE tree_id = :tree_id AND path LIKE :root_path || '%';

-- ltree 版
DELETE FROM tree_node WHERE path <@ :root_path;
```

如果保留了 `parent_id` 外键 + `ON DELETE CASCADE`，也可以只 `DELETE FROM tree_node WHERE id = :root_id` 让数据库自动级联（但级联是按邻接表走的，会更慢）。

### 4.3 重命名节点（路径段变化）

`ltree` 版可以用 `subpath` + 拼接重写整棵子树。字符串版同理，但要小心**子串替换的歧义**——`/12/` 不能被 `'1' → 'A'` 改成 `/A2/`。这就是为什么前后都加 `/` 当锚点的原因。

## 5. 性能分析

| 操作 | 字符串路径 | 数组路径（GIN） | `ltree`（GiST） |
| --- | --- | --- | --- |
| 取子树 | O(子树大小)，前缀索引 | O(子树大小) | O(子树大小)，且支持复杂过滤 |
| 取祖先 | O(深度)，需先拆字符串 | O(深度) | O(深度) |
| 取直接孩子 | 需 `depth + 1` 过滤 | 同 | 用 `nlevel` |
| 插入节点 | O(1) | O(1) | O(1) |
| 移动子树 | O(子树大小) | O(子树大小) | O(子树大小)，原生函数支持 |
| 索引体积 | 中（B-Tree on TEXT） | 大（GIN） | 中（GiST） |

**关键结论**：物化路径用"写入时多花一点钱 + 多存一列"换来了"取子树几乎免费"。这是它最大的卖点。

## 6. 典型陷阱

### 6.1 前缀冲突

如果路径写成 `'1/4/9'`（不带前后斜杠），`LIKE '1/4/%'` 会同时匹配 `'1/4/9'` 和 `'1/40/9'`。**永远在前后都加分隔符**。

### 6.2 ID 与 path 不同步

`path` 是冗余字段，代码里所有写入路径必须经过统一的 helper 来更新它，否则一定会出现"`parent_id` 指向 4，但 `path` 还是 `/1/3/`"这种诡异 bug。建议：

- 用触发器在 INSERT/UPDATE 时自动重算；
- 或者用 ORM 的"模型钩子"集中处理。

### 6.3 路径过长

字符串路径有最大长度限制（`VARCHAR(1024)` 通常够用，但深度 + 单段 ID 长度爆增的场景要小心）。预估上限：

```
最大深度 × (max(id_digit) + 1) + 1  ≤ 列长度
```

例如 `BIGINT` 最长 19 位，深度 30 就要 30 × 20 + 1 = 601 字符，要留出余量到 1024。

### 6.4 索引选择

- 字符串 `path` 上的 B-Tree 只能加速 **左前缀** 匹配（`LIKE 'xxx%'`），无法加速 `LIKE '%xxx%'`。
- 想加速"任意位置匹配"必须用 `pg_trgm` + GIN 之类的扩展索引。
- 数组路径用 GIN，可以快速判断"包含 / 被包含"。

### 6.5 不同 collation 导致的 LIKE 行为差异

某些数据库的字符串 collation 是 `*_ci`（不区分大小写），如果路径里夹杂大小写敏感的标签，`LIKE` 可能会出错。**最简单的办法是只用数字 + `/`**。

## 7. 优缺点

### 7.1 优点

- **取子树 / 取祖先都极快**：一次索引扫描搞定。
- **可读性好**：直接看 `path` 就知道在树里的位置。
- **`ltree` 提供了工业级生态**：是 PostgreSQL 的"杀手级"特性之一。

### 7.2 缺点

- **写入有放大效应**：移动子树 = 重写整棵子树。
- **冗余字段需要严格维护**：`parent_id` 和 `path` 必须同步。
- **路径长度有上限**：超深的树或超长 ID 都可能撑爆。
- **环检测仍然需要应用层 / 触发器**。

---

## 8. 适用场景

最适合用物化路径的场景：

1. **目录 / 类目树**：经常整棵子树取出来给前端渲染。
2. **PostgreSQL 项目**：直接上 `ltree` 几乎是默认答案。
3. **路径本身有业务含义**：例如 URL 路径、文件路径，本来就要保存原始字符串。
4. **读多写少**：移动子树的代价被读多次摊销。

不太适合的场景：

- **频繁拖拽 / 移动节点**的协作类编辑器（每次拖拽都要批量更新）。
- **超深树或极动态结构**：路径长度不可控。

## 9. 与 `tree.py` 的映射

| 内存模型 | 物化路径列 |
| --- | --- |
| `TreeNode.value` | `payload` |
| `TreeNode.name` | `name` |
| `TreeNode.parent` | `parent_id`（保留） + `path` 倒数第二段 |
| `TreeNode.children` | `WHERE path LIKE self.path \|\| '%' AND depth = self.depth + 1` |
| `TreeNode.level` | `depth`（直接对应） |
| `TreeNode.order` | DFS 编号本身就和 path 字典序高度相关，可派生 |
| `Tree.root` | `parent_id IS NULL` |

### 9.1 一个有趣的观察

`tree.py` 里的 `order` 是按 DFS 先序在叶子上**全局编号**。物化路径的 `path` **按字典序排序**也几乎等价于 DFS 先序遍历——只要在生成路径时给每段做合适的零填充（`'/001/004/009/'` 而不是 `'/1/4/9/'`），按 `path` 排序就能直接得到 DFS 顺序，连 `order` 列都可以省掉。

```sql
SELECT * FROM tree_node WHERE tree_id = :id ORDER BY path;
-- 等价于按 DFS 先序遍历整棵树
```

这是物化路径"一个字段顺便解决遍历顺序"的隐藏福利。

---

## 10. 一句话总结

> **物化路径用一列冗余字符串换来了"取子树 / 取祖先 O(1) 索引扫描"的性能。** PostgreSQL 用户应该直接上 `ltree`；其他数据库用字符串或数组路径，并务必把维护逻辑封装到触发器或 ORM 钩子里。
